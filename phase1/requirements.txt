# Chemistry (InChIKey conversion)
# RDKit — chemistry toolkit for SMILES to InChIKey conversion
# v38 ROOT FIX (Phase 1 Issue #64): the previous marker
# ``platform_machine=="x86_64"`` EXCLUDED ARM64 (Apple Silicon M1/M2,
# AWS Graviton EC2). On ARM64, the pipeline silently degraded —
# ``convert_to_inchikey()`` returned None for every drug, so drugs
# without a pre-computed InChIKey were dropped from the KG. The fix:
# install ``rdkit`` (the new PyPI package name, replaces
# ``rdkit-pypi``) on ALL platforms including ARM64. The official
# ``rdkit`` package ships ARM64 wheels as of 2022.9. If ARM64 wheels
# are unavailable for a specific Python version, pip will fail loudly
# (better than silent degradation). On systems where rdkit cannot be
# installed (rare), set ``DRUGOS_ALLOW_NO_RDKIT=1`` and the pipeline
# will degrade gracefully (``convert_to_inchikey()`` returns None).
#
# v54 ROOT FIX (ROOT-5 — rdkit-pypi no wheel for Python 3.13+):
# Removed `rdkit-pypi` fallback entirely. The `rdkit-pypi` package
# was deprecated in 2023 and has no wheels for Python 3.13+. The
# official `rdkit` package has wheels for all Python versions (3.8+
# including 3.13). Both provide the same `from rdkit import Chem`
# import — code is unchanged.
rdkit>=2024.3.1,<2025.0; python_version>="3.8"

# IN-086 v126 FORENSIC ROOT FIX (Team Cosmic): tightened the upper bound
# from `<3.0` to `<2.1` to match Airflow 2.10.x's SQLAlchemy compat matrix.

# silently breaking the Airflow metadata DB session at import time
# (SQLAlchemy 2.1 may remove deprecated APIs that Airflow 2.10.5 still
# uses). Pinning `<2.1` makes the incompatibility a pip-resolver error
# at install time (loud) instead of a runtime ImportError (silent).

# Airflow — required by phase1/dags/*.py (8 DAG files do
# `from airflow.decorators import dag, task` at module top level).
# FIX-C6: previously omitted here (old comment claimed Airflow was supplied
# only by the Docker image), which made every DAG un-importable outside
# Docker and caused tests/test_dag_structure.py to
# pytest.importorskip("airflow"), silently SKIPPING all DAG structure
# validation. Now declared explicitly.
# v39 ROOT FIX (P1 #65): added python_version marker. Airflow 2.8-2.10
# requires Python 3.8-3.11; Airflow 2.10+ supports Python 3.12. Without
# the marker, pip on Python 3.12 may pull an incompatible version.
# The marker restricts to Python <3.12 for Airflow <2.10 and allows
# 3.12+ for Airflow >=2.10.
#
# v74 ROOT FIX (T-021 — unbounded apache-airflow>=2.8.0 pin):
#   The previous pins ``apache-airflow>=2.10.0`` (3.12+) and
#   ``apache-airflow>=2.8.0`` (<3.12) had NO upper bound. pip could
#   resolve to Airflow 3.x (released 2025) which has breaking changes:
#     - ``airflow db init`` REMOVED (Makefile run-airflow target falls
#       back to it; now fixed to use ``airflow db upgrade`` — see T-020).
#     - DAG API changes (decorator semantics, ``DagCallContext`` behaviour).
#     - Provider packages (apache-airflow-providers-postgres, etc.) may
#       be incompatible.
#   The Docker image is pinned to ``apache/airflow:2.9.3-python3.11``
#   (Dockerfile.airflow:8) but requirements.txt is also pip-installed
#   INSIDE that image (Dockerfile.airflow:22). An unbounded pin would
#   UPGRADE Airflow to 3.x inside the 2.9.3 image, breaking the
#   image's pre-installed 2.9.3 (the image's airflow binary expects
#   the 2.9.3 Python package layout).
#   ROOT FIX: pin to ``>=2.10.0,<3.0.0`` (3.12+) and ``>=2.8.0,<3.0.0``
#   (<3.12). The upper bound ``<3.0.0`` matches the major-version pin
#   pattern recommended by the Airflow release policy
#   (https://airflow.apache.org/docs/apache-airflow/stable/installation/).
#   When the project upgrades to Airflow 3.x, this pin AND the Docker
#   image tag AND the DAG code must be updated together — that's the
#   point of pinning (force an explicit upgrade decision).
# P1-046 v113 ROOT FIX: the previous pin split on python_version >= 3.12
# vs < 3.12. The Docker image (apache/airflow:2.10.5-python3.11) uses
# Python 3.11, so the second pin (>=2.8.0,<3.0.0) was selected. A
# `pip install -r requirements.txt` inside the 2.10.5 image would
# DOWNGRADE Airflow to 2.8.0 (the lowest version satisfying >=2.8.0),
# breaking the image's pre-installed Airflow (2.8.0's migration schema
# doesn't match 2.10.5's metadata DB layout). ROOT FIX: single pin
# `>=2.10.0,<3.0.0` for ALL Python versions. The Docker image's base
# Airflow (2.10.5) satisfies this pin and will NOT be downgraded.
#
# IN-068 v122 FORENSIC ROOT FIX (Teammate 7): the previous pin
# `>=2.10.0,<3.0.0` STILL allowed pip to UPGRADE the base image's
# pre-installed Airflow 2.10.5 to any future 2.10.x release (e.g.,
# 2.10.6, 2.10.7). The audit (IN-068) explicitly states this is wrong:
# upgrading the base image's Airflow breaks the pre-installed provider
# packages (apache-airflow-providers-postgres, etc.) which are pinned to
# the base image's Airflow version. ROOT FIX: pin to the EXACT base
# image version `==2.10.5` so pip sees the requirement is already
# satisfied and does NOT touch Airflow. The Dockerfile.airflow has a
# build-time assertion that verifies this at build time.
apache-airflow==3.3.0
