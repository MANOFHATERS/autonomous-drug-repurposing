# FIX-C6: Airflow is a REAL dependency — phase1/dags/*.py (8 files) do
# `from airflow.decorators import dag, task` at module top level. Without
# apache-airflow installed, every DAG crashes at import time and
# phase1/tests/test_dag_structure.py used to pytest.importorskip("airflow"),
# silently SKIPPING all DAG structure validation. The Docker base image
# (apache/airflow:2.10.5-python3.11) includes it for prod, but dev/CI envs
# need it too.
#
# Task 364 ROOT FIX: the audit demanded pinning apache-airflow==2.10.5 to
# match the Dockerfile.airflow base image. We tried the literal pin
# `apache-airflow==2.10.5` + `sqlalchemy==2.0.30` (Airflow's official
# constraint pin) but pip's resolver STILL hit ResolutionImpossible
# because some transitive dep of Airflow 2.10.5 (one of its provider
# packages) requires a different SQLAlchemy version.
#
# ROOT FIX (production-correct): use the same major-version pin pattern
# as phase1/requirements.txt — `>=2.10.0,<3.0.0` for Python 3.12+ and
# `>=2.8.0,<3.0.0` for <3.12. This:
#   1. Prevents Airflow 3.x breakage (the audit's actual concern).
#   2. Allows pip to resolve transitive deps (provider packages) freely
#      within the 2.x range.
#   3. The CANONICAL production pin (apache/airflow:2.10.5-python3.11)
#      lives in the Dockerfile.airflow base image — that's the version
#      that ships to production. Dev/CI uses the looser pin so the test
#      suite can install without resolver conflicts.
# When the project upgrades Airflow 3.x, this pin AND the Docker image
# tag AND the DAG code must be updated together (per Airflow release
# policy: https://airflow.apache.org/docs/apache-airflow/stable/installation/).
apache-airflow>=2.10.0,<3.0.0; python_version>="3.12"
apache-airflow>=2.8.0,<3.0.0; python_version<"3.12"
